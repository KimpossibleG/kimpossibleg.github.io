Keeping it Classy

When I first stumbled into creating classes in Ruby, I was hard-pressed to find a definition of a class or an instance that didn't rely heavily on an example. I found "a class is the blueprint from which individual objects are created" on Tutorials Point, and "classes are the basic template from which object instances are created. A class is made up of a collection of variables representing internal state and methods providing behaviours that operate on that state." on WikiBooks. Re-reading them now, as someone who has played with classes for a bit, that lingo is kinda-sorta starting to make sense to me -- but the reality is, for non-technical people, those explanations are in a foreign and mysterious language! :-) I, too, will now join the ranks of those who rely on examples to clarify this 'class' business. Ready, fellow laymen?

Look around you, and mentally list three things you see that you use fairly often. My list? A sofa, a coffee table, and a dog toy. I bet you recognize these words. You may even have these items yourself, and you most likely know what you can do with them. For example, we know that a coffee table typically has a flat surface for holding things like coffee and magazines, that it is typically found in a living room type of area, and that we shouldn't sit on it or feed it to the dog. Having a name for this item saves us from having to describe it over and over and over again whenever we want to reference it. Classes in Ruby exist for the same reason. When we mention a certain class -- an array, a hash, a string -- those of us who speak Ruby automatically know what is being spoken about, and what kinds of things can be done with it -- i.e. what methods it has.

When we start getting creative in Ruby and building our own furniture or household items, we define them as classes, too -- and we initialize them so that they have the standard features we'd come to expect from that item. For example, suppose that, in Ruby, we're building an app to do our taxes. Taxes, bless them, have a zillion forms. Instead of writing code to create a form every single time we want to use one, we can create a class called Form that includes the common features we'll be using: a free text area, a check box, a spot for your signature. We'll also define things that we can do with that form, like print it, email it, or throw it away.

To keep things super straightforward for now, let's walk through a quick example. Suppose that we are going to be making checklists and checklists and more checklists in Ruby. It'd be wise to create a CheckList class, so that we can use a basic checklist whenever we have the need. We'd do that as follows:

class Checklist(number_of_items, title)

def initialize

@number_of_items = number_of_items

@title = title

end

end

That is now a basic blueprint for building a Checklist. When we create one, we'll specify the number of items we want on it, and its title, and voila - we have the outline of basic checklist.

What's with the @this = this business? That exists because every checklist that we create is going to need its own title. We won't, for example, want to call every one of them Checklist o' Cool. Because each one will have a title, and that title will be different, we say that this is an INSTANCE of a Checklist, with an INSTANCE VARIABLE of a title, and of a number of items. Defining them in our class with the @ symbols lets Ruby know that, every time we have a new checklist, we're going to have a new title and number of items.

Within every instance of a Checklist we create, we might want to do similar things -- like check off an item from the list. Within our class definition, we can create methods to do that, such as:

def check_off_item(@item_number)

if @item_number == "complete"

delete item_number

else

return "You haven't done this!"

end

Now, everytime we want to remove an item, we can call this method:

our_list.check_off_item(4)

...will remove item 4 from the list. No need to write the code every time!

While this is a very simplistic take on classes, which are actually quite powerful and allow us to do some awesome things in Ruby, the fundamental point is there: a class gives us a blueprint standard, and its instance variables and methods let us do specific things that make sense for the object.

How can cheap storage be troublesome for tech?

There wasn't a lot of waste and garbage back in the good ol' days. Then, along came the industrial revolution, and along with it came oodles and tonnes and mind-boggling amounts of waste, and it had to go somewhere. We solved that problem by digging a little hole and throwing it in. Then, we dug another hole. And another hole. And now, our landfills are all over the place, expensive, and causing significant problems to our world.

Enter the tech revolution. With the arrival of computers came the production of data. At first, it was just a little. Then a little more. Now, the amount of data that we have is astronomical -- and it's all gotta go somewhere.

The solution has been data storage. At first, it was reasonably expensive, but we didn't need a lot. As we needed more and more, it became less and less expensive. Awesome, right?! We have ended up with buildings and buildings full of storage. They need air conditioning. They emit CO2. And, sometimes, getting specific data out of them can be a giant -- and expensive -- pain in the rumpus. Just to add insult to injury, the amount of data that the world is producing -- and the inexpensive data storage solution -- means that these centers are becoming more and more prolific, and therefore more problematic.

So, what do we do? We approach the problem in a similar way to how we've been approaching our garbage problem. We need to reduce, figure out how to manage our data. There, the similarities between garbage and data (mostly) end.

One of the most simple solutions to our data problem could be just to delete it. Yes?

Technical complications aside, we're now living in a world where most people don't want to get rid of their data. They want those pictures, those videos, the songs -- and that's just personal stuff. Whole industries are changing the ways that they produce and use data. The medical industry, for example, is digitizing x-rays and patient records. Aviation? Manufacturing? They're creating incredibly complex (and data-heavy) experiments and simulators. The result is data, data, and more data -- not less. Let's throw common ol' business into the mix. Sometimes, they'd be (more than?) happy to delete their data (Enron and Arthur Andersen, anyone?), but that isn't morally or legally in their best interests. Still, do the emails about company luncheons and videos of the CEO dancing on a picnic table need to be saved -- especially if they've been sent to every member of the company? Hello, unnecessary and expensive storage!

Solutions? They're in the works. A group has come together called the Storage Networking Industry Association (SNIA). It's a not-for-profit association dedicated to connecting "the IT industry with end-to-end storage and networking solutions." The SNIA has defined the principles of information lifecycle management (ILM), which focusses on getting rid of the data that companies dont', and won't, need as quickly as possible, while maintaining the rest in a way that is cost-effective and accessible.

A lot of the rest is going to be about user behaviour... and that is going to be a lot like teaching people to reduce, reuse, and recycle. Takes time. A lot of time.


For more information, see this paper, by IBM. Advertising, yes -- but very informative.

Taking the Cycle-Path

Let's talk cycling.

We all know it hurts your butt. What you might not know is that geeks do it, whenever they want, with minimal butt-pain involved whatsoever. Yeah, that's right. And like a woman about to get thrown out of the magician's union, I'm going to show you what's behind the curtain.

It's a method in Ruby.

That's the end of the fun part. (To each her own, right?! ;-) ) Read on if you'd like to know how it works.

You call cycle on an array the standard way: array.cycle , and need an object and a block in curly braces. If you leave that last part out, and just call .cycle on the array, it will return the Enumerable.* The method can also take an argument. If left empty, it defaults to nil. Otherwise, it should be a positive integer. That integer will tell the method how many times to repeat -- or cycle -- the array through the block.

The object is any new variable, which represents each element of the array. The block tells us what to do with that element... puts it? print it? collect it? By pulling this altogether, we will take the array, repeat/cycle it the specified number of times, and then puts/print/etc it, as specified by the block. For example:

kids = ["Ali", "Eli", "Liz"]

kids.cycle(3) {|x| puts x }

=> ["Ali", "Eli", "Liz","Ali", "Eli", "Liz","Ali", "Eli", "Liz"]

As my mother-in-law likes to say: Winner, winner, chicken dinner. :-)

Hash, you array of sunshine!

What if I told you that every array is actually a hash?

It's true. Sort of.

Here's the deal: An array is just a big ol' numbered list, where the numbers are invisible. That's it, crew. The first number is 0, which is kinda screwy unless you grew up in one of those areas where the ground floor of a building is known as something other than the first floor, but I digress. Arrays = numbered lists. Ta-da!

A hash? Stick with me here. A hash is also a list. It can be a numbered list, if you like. Or, it can be a lettered list. Yep. Now, let's get a little cray-cray: It can a list listed by ANYTHING YOU WANT, as long as each of those things is unique. Examples? These are all valid hashes:

Example 1:

1. potato

2. car

3. Edward


Example 2:

A. potato

B. car

C. Edward


Example 3:

monkey. potato

blue. car

spoon. Edward


But this one? This isn't valid. Two of the indexes are the same. FAIL.

monkey. potato

blue. car

blue. Edward


So, that's it. Arrays are pretty much just the most basic form of hashes, always numbered, with invisible numbers. Cool?

If that's the case, why would we ever use an array? We'd use them whenever we want just a simple, simple list, that is invisibly numbered. For anything else, we'll go hash. They have a lot more flexibility, and can also act as mini-mini-databases. Dig it.

Pairing & Feedback

I really enjoy working with other people -- particularly smart, inventive, awesome people, like the ones that I have met "live" thus far in Dev Bootcamp: Big shout-outs to Catherine, Tyler, Xander, and Mikee! You guys rock!

Annnnnd, I hate pairing. I loathe it. I feel like a useless knob when I am acting as somebody else's fingers (aka being the Driver) and I feel stressed, rushed, and incompetent as the Navigator. It's a real struggle for me to be learning in either case. I feel that the uber-structured format of pairing is extremely limiting and hinders creativity and helpful conversation. It has been in the cases where my pair and I stray from the offical model and actually talk to each other and throw ideas around that I feel like we're both really benefitting -- and then we lose that momentum when we slip back into who is going to be driving and who is going to be navigating. Perhaps pairing will be more comfortable and productive when I know my pair well, know his/her level of skills and how it compares to my own, know his/her idiosyncracies, etc.

The peer feedback that I've received on my coding and challenges thus far has been of excellent quality. It has consistently been specific, actionable, and kind -- and really helpful. I feel great getting the feedback!

I haven't received any written feedback via the Feedbackinator yet, so I can't comment on how it will feel to get anonymous feedback. I have noticed, however, that the biggest gap in giving feedback (based on all the comments that I have read and rated) is when it comes to giving Actionable points. Everyone seems to quick to tell each other how fabulous they are, or how much they enjoyed the pairing session, but very few seem to provide suggestions for improvement. I find that disappointing.

I am looking forward to getting some feedback from DBC staff, and confess that I am surprised that I haven't received any yet -- at least not that I know of! I know that I can get my code to pass tests, but I think that is only half the battle. Does my work indicate that I am 'getting it,' or am I relying on band-aids and workarounds too much? Is my pseudocode strong, and would it be helpful/appropriate for a work environment? I've been veering from the reflections guidelines and focussing on what is most helpful for myself. Am I still gaining from the exercise in the way that DBC intended?

Writing feedback has been pretty painless. I do find that I am a bit wordy, but I don't think that negatively impacts the effectiveness of my messages. I do make an effort to be specific, actionable, and kind -- and it's great to have those guidelines in place.

JavaScript, the Popular. Why do we love thee?

I've been asked why JavaScript is so darn popular.

My answer? ... "JavaScript is popular?!"

This whole JavaScript thing is pretty new news to me, but now that I'm working on becoming a fancy coder-pants, I suppose it's good to know the answer. In fact, it's probably good to know what the eff it is.

So.

Umm... :-)

JavaScript is a programming language. We use it to make programs. Right? Right. But, more fundamentally, it makes web pages more dynamic. So, is it a replacement for HTML (which is the language we use to make webpages look like webpages instead of like pages of code)? No, sirree. HTML is still the essence and building blocks of webpages; JavaScript makes it do stuff. An analogy? HTML is like printed song lyrics; JavaScript is what takes those lyrics and turns them into a karaoke video. Before JavaScript, HTML just sat around on the web and looked pretty. With the addition of JavaScript, links now change colors when you hover over them. Pictures can scroll. You can click on a button or an image and it will change. That, my friends, is because of our good pal JavaScript.

What's not to love? Back in the days of Netscape, who created JavaScript, JavaScript and the interwebs were like drunk freshman at a party: alllllll over each other, and confident that it was true love! Then, (dum dum dummmmmm!), along came some better looking kids who may have even been sober and interesting and just plain better, like Shell, PHP, Perl, and Ruby. Still, JavaScript has become and remains the absolute most popular web dev language out there. What gives?

Ubiquity: That's what gives. All the main browsers out there support JavaScript. As well, the libraries and extra tag-along bits and pieces that go along with JavaScript have become HUGE -- which is akin to a fully, incredibly, mesmerizingly well stocked kitchen to someone advised to make a meal. If all you have is carrots and potatoes, a pot, and a hotplate, there's a good chance you're having boiled carrots and potatoes, The End. In a kitchen with every imaginable slicer, dicer, ingredient, spice, utensil, and thinga-ma-jig, a gourmet spread is possible. Who doesn't want a gourmet website instead of carrots and potatoes?

Add to all of this that JavaScript is relatively easy to learn, plays well with DOM, and processes on the client side, and we have a winner. The interwebs has a winner. And that, my friends, is why JavaScript is the most popular girl at the party.

Exploring My Values

Three times in my life when I recall being my absolute happiest, most satisfied, and proudest include my wedding day, watching my young daughter on her first carnival ride, and wrapping up a speech at a conference. The values that come to mind when reflecting on these occasions have quite a bit of overlap:

  • Close relationships
  • Confidence
  • Excitement
  • Introspection
  • Enthusiasm

The value of those that is particularly important to me is Introspection. I think that a false happiness is easily achieved, perhaps through adrenalin or celebration, or the use of drink or drugs. When introspection is involved, however -- when one can look inside and feel, understand, accept, and truly believe that the right choices are being made, that growth is happening, or that something is genuinely beautiful to you -- THAT'S when true happiness and satisfaction is possible.

On a scale of 1-5 where 1 means Disagree and 5 means Agree, this is how much I try to live up to these values:

  • Close relationships: 5
  • Confidence: 4
  • Excitement: 4
  • Introspection: 5
  • Enthusiasm: 4

Close relationships and introspection are two of the most important elements of who I am and how I live my life. While I appreciate confidence, excitement, and enthusiasm, I feel that confidence and excitement are more the result of what I'm doing than values, and that enthusiasm is a value that I must consciously apply whenever appropriate.

The last topic that someone asked me for advice on was parenting -- specifically, how and when to go about having a toddler give up her pacifier. I've been a parent for almost four years, and it's been an ongoing shock to me that I'm actually pretty decent at it. I never thought I'd have the magical parenting touch. The fact that my friends ask me for advice on various elements of parenting -- and then often go on to accept and apply the advice -- makes my heart grow six sizes every time. I suppose that makes sense: Turns out that I value close relationships and introspection, which is a lot of what parenting is all about!

Three Websites

The three websites I assessed for this article are:

  • thebloggess.com
  • news.google.com
  • buzzfeed.com

Let's start with The Bloggess.

This is one of my favorite sites simply because I enjoy the author's humour. She's irreverent, honest, funny, and a little nuts -- kind of like me, in some regards!

As much as I'm not a fan of the color pink, I like the theme of this website. I find that the right things pop out at me -- the article titles, for example. The opening image in the header is both memorable and unique. What I don't like about this page is its flow between articles. When I read blogs, I tend to catch up on weeks or days worth of entries all at once, and it's important to me that I can do so fluidly. In this blog, I have to dig back several articles using the Previous link, read my way forward, working from the bottom of the page up. This makes me more insane than I find enjoyable.

The author of this blog has quite a bit of stuff for sale, and I've not bought any. I have, however, been entertained by it. I've also purchased her book, but I did so at a real-life bookstore. I bought it there because I wanted it in a hurry and didn't want to wait for shipping.

The next site I looked at was news.google.com.

I spend quite a bit of time on this site, and I've developed definite preferences for which device I use to read it. My smartphone and laptop are definitely winning out over my tablet. I think I enjoy the smartphone version because it contains only the bare necessities for reading through it, including simple arrows to access more articles/content, and a very limited number of articles per section (3). There is quite a bit more clutter on the laptop version, but I'm okay with that, because I have the bigger screen and can scroll quickly. The tablet, however, has combined the worst elements, in my opinion. Instaed of simple arrows to expand, there is an 'expand content' button that opens up to a bunch of different sources and articles, instead of a nice, compact list. As well, the tablet version shows me most popular and trending articles on the bottom right, and then suggestions by news source on the bottom right, and I have to scroll by news source. This is, for me, both irritating and a waste of time.

I feel like my time on this site is well-spent, because I'm (supposedly) learning about the world and things that ought to make me a better-versed, more well-rounded person. Doesn't feel quite so awesome when I catch myself reading about Bieber and Rob Ford -- AGAIN.

Buzzfeed...

...is what I turn to when I need to pass some time and I've read all the news and finished checking facebook. ;-) Buzzfeed is typically a total waste of time, and I feel like I need to wash after reading it. I usually view it on my phone or tablet, and the navigation is dreadful... After reading any of the articles, paging back brings me back to the top of the buzzfeed list o' crap, and I have to scroll allllllll the way down to find new content. This really ought to discourage me from continuing to read. It doesn't. That adds to my feeling that I am wasting my life.
